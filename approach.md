# 工作模式

本项目由用户通过自然语言指令、Claude Code CLI 执行全部代码工作。以下是每次用户提出意见后的标准工作流程。

## 标准流程（按顺序执行）

### 1. 分析问题，确认根因

- 阅读相关代码，理解当前实现
- 如果是 bug，先找根本原因，不猜测，不盲目修改
- 如果是新功能，分析影响范围和最简实现方案
- 如有必要，与用户确认方案后再动手

### 2. 修改代码

- 只改动与问题直接相关的部分，不顺手重构
- 修改完成后做静态验证（`python3 -c "import ast; ast.parse(open(...).read())"`）
- 如有多个修改点，逐一确认每处都正确

### 3. 请用户确认

- 向用户说明做了什么、为什么这样做
- 等待用户确认问题已解决，再进行下一步

### 4. 更新 WORK_NOTES.md

- 记录本次变更的技术细节：改了什么、为什么、设计决策
- 格式：`- **YYYY-MM-DD** — 简短标题：详细说明`
- 追加到变更历史末尾，不覆盖已有内容

### 5. 更新 user.md

- 将用户本次的原始消息追加到文件末尾
- 格式：`## Message N [YYYY-MM-DD]\n\n用户消息内容\n\n---`
- 消息原文照录，不修改、不翻译

### 6. Git 提交

- `git add` 涉及的文件（`scholar_citation.py`、`WORK_NOTES.md`、`user.md`，以及其他改动文件）
- commit message 用英文，说明做了什么、为什么，不只写"what"，也写"why"
- 每次只提交一个逻辑单元，不把多个不相关的修改混在一起

---

## 调试原则

- **不猜测**：先读代码、找根因，再改
- **不叠加修复**：一次只改一个假设，确认有效再继续
- **失败三次以上**：停下来，质疑架构，和用户讨论

## 代码修改原则

- 最小改动：不改与问题无关的代码，不加多余注释、类型注解、错误处理
- 不创建不必要的新文件
- 不引入新依赖，除非必要

## 文档原则

- `WORK_NOTES.md`：技术细节、架构决策、踩坑记录，供自己和贡献者参考
- `user.md`：用户原始输入历史，展示 AI 辅助开发的完整对话轨迹
- `README.md`：面向外部用户的功能说明，有重大功能更新时同步更新
- `approach.md`：本文件，描述工作流程本身

## 特殊情况

| 情况 | 处理方式 |
|------|---------|
| 用户要撤销上次提交 | `git reset HEAD~1`，修改后重新提交 |
| 需要测试但网络不可用 | 做静态语法检查 + 代码逻辑审查，标注"待网络恢复后验证" |
| 用户打断正在进行的操作 | 停止当前工作，处理新指令，再决定是否回到原任务 |
| 发现隐私信息 | 立即替换为占位符，检查 git 历史是否也需要清理 |
